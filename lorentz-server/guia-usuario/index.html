    Guia do Usuário — Lorentz Server¶

    1. Introdução¶

    O Lorentz Server é o serviço responsável por conectar a central telefônica ao sistema de atendimento. Ele roda em segundo plano no servidor e gerencia:

    • Distribuição de chamadas — encaminha ligações da fila para agentes disponíveis
    • Controle de agentes — login, logout, pausas e estados de atendimento
    • Monitoramento — fornece informações em tempo real para o Supervisor
    • Integração com URA — recebe dados de navegação da URA para exibição na tela do agente

    2. Requisitos do sistema¶

    Requisito Especificação
    Sistema operacional Windows Server 2016+ ou Linux (com .NET 9)
    Runtime .NET 9.0
    Banco de dados SQL Server 2016+
    Central telefônica 3CX, Asterisk ou Intelbras Impacta
    Memória RAM 2 GB mínimo
    Porta de integração TCP 8001 (padrão) — conexão com aplicativo do agente
    Porta de serviço TCP 3000 (padrão) — integração com Supervisor
    Porta URA TCP 4000 (padrão) — integração com IVR
    Porta de monitoramento UDP 8000 — comunicação com Monitor CTI

    3. Instalação¶

    3.1 Primeira instalação¶

    1. Copie os arquivos do Lorentz Server para a pasta de destino (ex: C:\Lorentz\ no Windows ou /var/lorentz/ no Linux)
    2. O arquivo de configuração .ini será criado automaticamente na mesma pasta do executável na primeira execução
    3. Configure os parâmetros conforme a Seção 4

    3.2 Executando o serviço¶

    O Lorentz Server funciona como um serviço em segundo plano (Background Service). Após iniciado, ele apresenta no console:

    • Versão do servidor
    • Endereço do banco de dados
    • Nome do banco de dados e usuário
    • Porta de integração configurada
    • Endereço da central telefônica

    4. Configuração¶

    Todos os parâmetros ficam no arquivo .ini ao lado do executável, na seção [Parametros]. A tabela abaixo descreve cada configuração com seu valor padrão.

    4.1 Conexão com o banco de dados¶

    Configuração Descrição Padrão
    Servidor SQL Endereço IP ou hostname do SQL Server 127.0.0.1
    Banco de dados Nome da base de dados LorentzCTI
    Usuário SQL Usuário de acesso ao banco (em branco)
    Senha SQL Senha de acesso (armazenada criptografada) (em branco)
    Driver SQL Driver ODBC utilizado {SQL Server}

    4.2 Portas de comunicação¶

    Configuração Descrição Padrão
    Porta de integração Porta TCP onde agentes e supervisores se conectam 8001
    Porta de serviço (Concentre) Porta TCP para consultas do Supervisor web 3000
    Porta URA Porta TCP para integração com o sistema de URA 4000
    Host TCP Endereço IP de escuta (se em branco, escuta em todas as interfaces) (em branco)

    4.3 Central telefônica¶

    Configuração Descrição Padrão
    Endereço da central (IP) IP da central telefônica (aceita múltiplas centrais separadas por vírgula) 127.0.0.1
    Porta da central Porta de conexão com a central 7001
    Fabricante Tipo de central (Intelbras, 3CX) Intelbras
    Modelo Modelo da central (Intelbras, 3cx) 1

    Múltiplas centrais: para conectar a mais de uma central, separe os endereços por vírgula. Cada central pode ter porta específica no formato IP:Porta (ex: 192.168.1.10:7001,192.168.1.20:7002).

    4.4 Funcionalidades¶

    Configuração Descrição Padrão
    Monitorar ramais Ativa monitoramento automático de todos os ramais Desativado
    Atender tronco Permite atendimento de chamadas diretas no tronco Ativado
    Mensagem de fila Ativa reprodução de mensagem para chamadas em fila Desativado
    Quantidade de fila Número máximo de chamadas simultâneas na fila 20
    Ramal de controle Ramal utilizado para operações de controle do sistema 101
    DISA Habilita atendimento automático DISA 0
    CallBack Dialing Habilita retorno automático para celulares (em branco)
    Ramal de fila Impacta Ramal utilizado como ponto de entrada da fila (Intelbras) (em branco)

    4.5 Gravação de chamadas¶

    Configuração Descrição Padrão
    Gravar tronco Ativa gravação automática de chamadas em troncos Desativado
    Modelo do gravador (1, 2, 3) Identificação do modelo de gravador para cada placa (em branco)
    Tronco do gravador (1, 2, 3) Troncos associados a cada placa de gravação (em branco)
    OGM Número da mensagem OGM (On-hold Greeting Message) 1
    OGM ativo Habilita reprodução automática de OGM Desativado

    4.6 Integração com Asterisk¶

    Configuração Descrição Padrão
    Asterisk Habilita integração com Asterisk (AMI) 0 (desativado)
    Username Usuário AMI para conexão com Asterisk (em branco)
    Secret Senha AMI (armazenada criptografada) (em branco)

    4.7 Registros de log¶

    Configuração Descrição Padrão
    Gravar log de eventos Registra eventos da central (CSTA) em arquivo Desativado
    Gravar log de pacotes Registra comunicação com agentes em arquivo Desativado
    Gravar log de erros Registra erros do sistema em arquivo Desativado

    5. Operação do dia a dia¶

    5.1 Inicialização¶

    Ao iniciar, o servidor:

    1. Carrega as configurações do arquivo .ini
    2. Abre as portas de escuta (integração, serviço, URA)
    3. Conecta-se à(s) central(is) telefônica(s)
    4. Registra no banco que houve reinício (para limpeza de estados)
    5. Inicia o ciclo contínuo de processamento (1 vez por segundo)

    5.2 Ciclo de processamento¶

    A cada segundo, o servidor executa em sequência:

    1. Escuta novos serviços — aceita conexões do Supervisor web
    2. Escuta novos agentes — aceita conexões do aplicativo do agente
    3. Escuta agentes web — verifica novas sessões via banco de dados (sem TCP)
    4. Recebe pacotes de agentes web — processa comandos de agentes web
    5. Escuta URA — aceita conexões do sistema de URA
    6. Recebe pacotes dos agentes — processa comandos de agentes desktop
    7. Recebe pacotes do monitor — processa solicitações de monitoramento
    8. Recebe pacotes da URA — processa XML de screen pop e pesquisa
    9. Processa centrais — verifica eventos de cada central conectada
    10. Distribui chamadas — encaminha chamadas em fila para agentes livres
    11. Atualiza fila de espera — envia status da fila para supervisores e agentes

    5.3 Distribuição automática de chamadas (ACD)¶

    O servidor distribui chamadas usando um fluxo de serviço configurado no banco de dados. Para cada chamada na fila, o sistema:

    1. Consulta o fluxo de roteamento do serviço
    2. Avalia cada etapa do fluxo (condições de horário, dia, DDD, etc.)
    3. Localiza um agente livre no grupo designado
    4. Prioriza agentes com receptivo prioritário ativado
    5. Se não houver agente prioritário, seleciona o agente livre há mais tempo
    6. Se nenhum agente estiver disponível, aplica transbordo (tempo de espera máximo antes de passar para próxima etapa)
    7. Transfere a chamada para o agente selecionado

    5.4 Tipos de roteamento no fluxo¶

    Tipo Descrição
    Desvio para grupo Encaminha para um agente livre do grupo especificado
    Desvio para agente Encaminha para um agente específico
    Desvio para ramal Encaminha para um ramal fixo (ex: recepção)
    Desvio para grupo de ramais Distribui em rodízio entre ramais do grupo
    Condição de data/hora Avalia se o horário atual atende a uma condição (ex: horário comercial)
    Intervalo de data/hora Avalia se o momento atual está dentro de um período
    Condição de DDD Verifica o DDD de origem da chamada
    Condição de telefone Verifica o número de origem da chamada
    Dia da semana Verifica o dia da semana atual
    Desligar chamada Encerra a chamada (ex: fora do horário de atendimento)
    Ignorar chamada Ignora a chamada sem processamento

    5.5 Fila de espera¶

    O servidor gerencia a fila de espera ordenando chamadas por hora de entrada (a mais antiga é atendida primeiro — FIFO). Informações da fila são enviadas automaticamente:

    • Para supervisores: quantidade de chamadas e tempo de espera por serviço
    • Para agentes: quantidade total de chamadas e tempo de espera dos serviços associados ao grupo

    6. Funcionalidades para o agente¶

    6.1 Login e autenticação¶

    O agente se conecta informando:

    • Usuário — identificação no sistema
    • Senha — autenticação
    • Grupo — grupo de atendimento
    • Ramal — ramal telefônico a ser utilizado
    • Tipo — agente ou supervisor

    Ao efetuar login, o servidor retorna: DDD local, operadora padrão, caminho de gravação, permissões e versão do servidor.

    6.2 Estados do agente¶

    Estado Descrição
    Login Agente acabou de entrar no sistema
    Pausa desligada Agente disponível para receber chamadas
    Pausa eventual Agente em intervalo pessoal (ex: banheiro, café)
    Pausa produtiva Agente em atividade que não envolve atendimento (ex: treinamento)
    Logout Agente saiu do sistema

    6.3 Estados do ramal¶

    Estado Descrição
    Livre Ramal desocupado, pronto para receber chamadas
    Discando Agente está discando um número
    Chamando Ramal está tocando para outro ramal ou número externo
    Tocando Ramal recebendo uma chamada
    Ocupado (interna) Em conversação interna
    Ocupado (entrante) Em conversação com chamada recebida
    Ocupado (sainte) Em conversação com chamada feita pelo agente
    Consulta Em consulta a outro ramal durante atendimento
    Conferência Em chamada com 3 ou mais participantes
    Retido Chamada em espera (hold)
    Inapto Ramal com falha de comunicação
    Bloqueado Ramal bloqueado pelo sistema

    6.4 Operações telefônicas disponíveis¶

    Operação Descrição
    Fazer chamada Disca para um número externo ou ramal interno
    Atender chamada Atende uma chamada recebida
    Desligar Encerra a chamada atual
    Reter chamada Coloca a chamada em espera
    Recuperar chamada retida Retoma uma chamada que estava em espera
    Transferir Encaminha a chamada para outro ramal
    Consulta Faz uma consulta a outro ramal sem desligar a chamada atual
    Conferência Junta a chamada retida com a consulta em uma conversa a três
    Captura de chamada Captura uma chamada tocando em outro ramal
    Enviar DTMF Envia tons DTMF durante uma chamada (ex: navegar em URA externa)

    6.5 Comunicação interna (chat)¶

    Agentes podem enviar mensagens de texto entre si. O sistema verifica permissões de chat entre grupos e agentes antes de entregar a mensagem.

    6.6 Alerta ao supervisor¶

    O agente pode acionar um alerta ao supervisor para solicitar ajuda. O alerta é enviado a todos os supervisores logados e aparece como notificação no painel de monitoramento.

    6.7 Gravação de áudio¶

    O sistema suporta gravação de chamadas com:

    • Gravação automática — configurada por grupo/agente
    • Gravação manual — agente escolhe motivo e inicia a gravação
    • Motivos de gravação — lista de motivos configuráveis no banco de dados

    6.8 Sincronização de horário¶

    O aplicativo do agente pode solicitar a data e hora do servidor para manter sincronismo, garantindo precisão nos registros de atendimento.

    6.9 Alteração de senha¶

    Agentes podem alterar sua própria senha diretamente pelo aplicativo.

    7. Funcionalidades para o supervisor¶

    7.1 Monitoramento de agentes¶

    O supervisor recebe em tempo real informações de cada agente:

    • Usuário, grupo e ramal
    • Estado atual (livre, em atendimento, pausa)
    • Horário de login
    • Tempo em cada estado
    • Contadores de chamadas (internas, externas, ativas, receptivas)
    • Número de telefone do interlocutor atual
    • Serviço em atendimento

    7.2 Discagem pelo supervisor¶

    O supervisor pode discar para qualquer ramal ou número externo a partir da sua estação, incluindo:

    • Chamada interna para ramal
    • Chamada externa com rota de saída
    • Desconexão de chamada

    7.3 Distribuição manual pelo supervisor¶

    O supervisor pode distribuir manualmente uma chamada em fila para um ramal específico, informando o código da chamada e o ramal de destino.

    7.4 Lista de ramais e agentes¶

    O supervisor pode consultar:

    • Lista de ramais — todos os ramais com descrição e estado atual
    • Lista de agentes — todos os agentes logados com ramal, estado do ramal e estado do agente

    8. Integração com URA¶

    O Lorentz Server se integra com a URA (Unidade de Resposta Audível) via protocolo XML sobre TCP. A URA pode:

    8.1 Screen Pop (exibição na tela do agente)¶

    A URA envia dados coletados durante a navegação (ex: CPF digitado, opção do menu) que serão exibidos na tela do agente ao receber a chamada.

    8.2 Pesquisa de chamada¶

    A URA pode consultar o código de registro de uma chamada ativa em determinado ramal.

    8.3 Resposta da URA¶

    A URA informa o resultado da navegação do cliente (opções escolhidas, dados digitados) que são associados ao registro da chamada.

    9. Monitoramento externo (Monitor CTI)¶

    O sistema suporta um aplicativo de monitoramento externo via UDP na porta 8000. Funcionalidades:

    Comando Descrição
    Conectar Registra o monitor e recebe versão do servidor, lista de agentes e ramais
    Desconectar Remove o monitor da lista de distribuição
    Consultar parâmetros Recebe todas as configurações atuais do servidor
    Alterar parâmetros Envia novos valores de configuração para o servidor (aplicados imediatamente)
    Status do sistema Verifica se o servidor está operacional
    Reiniciar centrais Reconecta todas as centrais telefônicas
    Atualizar licença Recarrega informações de licenciamento

    10. Logs e diagnóstico¶

    O servidor gera os seguintes arquivos de log na pasta do executável:

    Arquivo Conteúdo Quando é gerado
    Erro + data + .log Erros do sistema com módulo, local e mensagem Quando "Gravar log de erros" está ativado
    Agente + data + .log Pacotes trocados entre servidor e agentes Quando "Gravar log de pacotes" está ativado
    Eventos .log Eventos recebidos da central telefônica (CSTA) Quando "Gravar log de eventos" está ativado
    LogCsta + data + .log Comunicação detalhada com a central Quando "Gravar log de eventos" está ativado
    LogURA + data + .log Mensagens XML trocadas com a URA Quando "Gravar log de eventos" está ativado
    Evento + data + .log Eventos gerais do sistema Quando "Gravar log de eventos" está ativado

    Dica: ative os logs apenas durante diagnóstico de problemas, pois o volume de dados pode ser significativo em operações com muitos agentes.

    11. Solução de problemas¶

    Problema Causa provável Solução
    Servidor não inicia Banco de dados inacessível Verificar endereço, usuário e senha SQL no arquivo .ini
    Agentes não conectam Porta de integração bloqueada Verificar firewall para a porta TCP configurada (padrão 8001)
    Chamadas não são distribuídas Fluxo de serviço não configurado Cadastrar fluxo de roteamento no Supervisor
    Chamadas ficam na fila Nenhum agente com receptivo ativo Verificar se agentes estão com pausa desligada e receptivo habilitado
    Mensagem "Problemas na conexão com Banco de Dados" SQL Server indisponível Verificar conectividade com SQL Server e credenciais
    URA não envia dados Porta URA incorreta Confirmar que a porta URA (padrão 4000) está liberada
    Monitor não recebe dados Porta UDP bloqueada Verificar se porta UDP 8000 está acessível

    12. Boas práticas¶

    1. Mantenha o banco de dados no mesmo segmento de rede do servidor para minimizar latência
    2. Configure apenas um servidor CTI por base de dados para evitar conflitos de distribuição
    3. Ative logs apenas para diagnóstico — desative após resolver o problema
    4. Monitore o serviço — se o servidor parar, todos os agentes perderão conexão
    5. Faça backup do arquivo .ini antes de alterar configurações
    6. Use transbordo nos fluxos — configure tempo máximo de espera por etapa para garantir que chamadas não fiquem presas na fila
